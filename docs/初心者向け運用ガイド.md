# EDINET Pipeline 初心者向け運用ガイド

## 1. このプログラムでできること
このプログラムは、EDINET の XBRL を読み込み、決算分析テンプレート Excel に自動で転記するバッチです。  
実行結果として、会社ごとの Excel、処理結果一覧 CSV、raw_edinet 監査用シートが出力されます。

---

### 開発場所
- `C:\Users\silve\EDINET_Pipeline`

### 実データ置き場
- `D:\EDINET_Data`

### 入力ZIPフォルダ
- `D:\EDINET_Data\input\zip`

### 出力先
- `D:\EDINET_Data\output\実行日時\`
- 実行するたびに、その時刻のフォルダが1つ作られます。

## 3. INFO / DEBUG の切り替え方法

### 切り替えるファイル
- `C:\Users\silve\EDINET_Pipeline\src\edinet_pipeline\config\settings.py`

### 切り替える場所
`LOG_MODE` を変更します。

### 通常運用
```python
LOG_MODE = "NORMAL"   # "NORMAL" or "DEBUG"
```

### 原因調査
```python
LOG_MODE = "DEBUG"   # "NORMAL" or "DEBUG"
```

### 動作

#### NORMAL のとき
- `LOG_LEVEL = "INFO"`
- worker の詳細ログは抑制されます。

#### DEBUG のとき
- `LOG_LEVEL = "DEBUG"`
- worker の詳細ログも出ます。

---

## 4. どういうときに INFO / DEBUG を切り替えるのか

### NORMAL でよいとき
- いつもの本番運用
- 成功 / 失敗だけ確認できればよいとき
- CSV と出力 Excel が問題なく作られているとき

### DEBUG に切り替えるとき
- 数値が入らない
- `file1 / file2 / file3` の割り当てが怪しい
- `raw_edinet` を見ても原因が分からない
- `failed_jobs.csv` だけでは原因が足りない
- NORMAL ログだけで原因が分からないとき

### 運用ルール
- 通常は `NORMAL`
- 原因調査時だけ `DEBUG`
- NORMAL ログだけで原因が分からない場合は、`DEBUG` に切り替える

## 5. file1 / file2 / file3 には何のファイルを入れるのか

このプログラムは、会社ごとに `file1 / file2 / file3` を自動で決めます。

### 半期あり会社
- `file1` = 半期報告書の最新1本
- `file2` = 有価証券報告書の最新1本
- `file3` = 有価証券報告書の次に新しい1本

### 半期なし会社
- `file1` = 有価証券報告書の最新1本
- `file2` = 有価証券報告書の次に新しい1本
- `file3` = 有価証券報告書のその次に新しい1本

### 実処理の読み込み順
- `parse_half_doc`
- `parse_latest_annual_doc`
- `parse_old_annual_doc`

の順で処理されます。

### 見分け方
ファイル名に次の文字列が入ります。
- 半期報告書: `jpcrp040300`
- 有価証券報告書: `jpcrp030000-asr`

### 注意
- `file2` と `file3` は、必ずしも「1期前」「2期前」とは限りません。
- このプログラムは、入力ZIPの中にある対象XBRLを新しい順に並べて選びます。
- そのため、手元に入れたXBRLの並び方によっては、1期飛ぶことがあります。
- 1社分として必要な本数が足りないと、その会社の job は作られません。

### 主な出力フォルダ

## 7. `D:\EDINET_Data\output\実行日時\` に入っているファイルの意味

### 7-1. `excel` フォルダ
会社ごとの完成版 Excel が入ります。  
ファイル名は次の形です。

```text
証券コード_会社名_期末日.xlsm
```

例:

```text
2206_江崎グリコ株式会社_2025-06-30.xlsm
```

### 7-2. `reports\batch_summary.csv`
全会社の処理結果一覧です。

#### 主な列
- `slot` = 何社目か
- `company_code` = 証券コード
- `company_name` = 会社名
- `status` = 全体結果
- `stock_status` = 株価書込結果
- `failure_reason` = 失敗理由
- `output_excel` = 出力Excelのパス

#### 使い方
- まず最初に見るファイル
- 成功したか失敗したかを最初にここで確認する

### 7-3. `reports\company_jobs.csv`
その回で各会社にどの XBRL を `file1 / file2 / file3` として割り当てたかを見るファイルです。

#### 主な列
- `has_half` = 半期ありか
- `file1`
- `file2`
- `file3`

#### 使い方
- 期待した会社・年度のファイルが選ばれているか確認する
- `file1 / file2 / file3` の割当確認はこのCSVを見る

### 7-4. `reports\failed_jobs.csv`
失敗・一部失敗・スキップされた会社だけを出すファイルです。

#### 対象
- `failed`
- `partial_success`
- `skipped`

#### 使い方
- 失敗した会社だけをすぐ探したいときに見る
- 空でもヘッダだけ作られることがある

### 7-5. `work` フォルダ
テンプレートからコピーした作業中 Excel が入る場所です。  
ファイル名には `_work_` が付きます。

#### 使い方
- 通常は最終成果物ではない
- 基本的には `excel` フォルダ内の完成版を見る

### 7-6. `logs\run.log`
通常ログです。  
実行開始、会社ごとの進行、CSV出力、終了までが入ります。

#### 使い方
- 実行全体の流れを確認したいとき
- NORMAL で分からない場合は DEBUG に切り替える

### 7-7. `logs\loop_summary.jsonl`
1社ごとの詳細サマリです。

#### 入る内容の例
- `raw_rows`
- `excel_ranges`
- `skipped_in_loop`
- `loop_total sec`

#### 使い方
- どの会社で何件書けたかを見る
- 詳細な調査に使う

## 9. Excel ファイルの `raw_edinet` シートの見方

`raw_edinet` は、何のタグが採用されたか、何が missing かを確認する監査用シートです。  
数値が想定どおり入らないときは、まずここを見ます。

### 列の意味
- `company_code` = 証券コード
- `doc_id` = 元のXBRLファイル名
- `doc_type` = `annual` または `half`
- `consolidation` = 連結 / 個別
- `metric_key` = 指標名
- `time_slot` = `YTD` または `Quarter`
- `period_start` = 期間開始日
- `period_end` = 期間終了日
- `period_kind` = `duration` または `instant`
- `value` = 元データ値
- `unit` = 単位
- `tag_used` = 採用したタグ
- `tag_rank` = 採用順位
- `status` = `parsed` / `OK` / `MISSING` など
- `run_id` = 実行ID
- `source_file` = 元ファイル名

### 見るポイント1
`metric_key` を見て、目的の指標があるか確認する。  
例:
- `NetSales`
- `OperatingIncome`
- `ProfitLoss`
- `TotalAssets`

### 見るポイント2
`tag_used` を見て、どのタグが採用されたか確認する。

### 見るポイント3
`status` を見る。
- `parsed` / `OK` = 取得できた
- `MISSING` = その行は空欄扱い

### 見るポイント4
`doc_type` と `time_slot` を組み合わせて見る。
- `half` + `YTD`
- `half` + `Quarter`
- `annual` + `YTD`

### 見るポイント5
`annual` の `YTD / MISSING` は、監査行として入ることがあります。  
これだけで不具合とは限りません。

## 10. プロの観点で入れておくべき補足事項

### 10-1. テンプレート Excel は直接編集しない
プログラムはテンプレートを `work` フォルダへコピーしてから書き込みます。  
安全のため、テンプレート本体に直接保存しない運用です。

### 10-2. NamedRange ベースで書き込んでいる
セル番地固定ではなく NamedRange に書いています。  
テンプレート修正時は、セル位置より NamedRange を壊さないことを優先してください。

### 10-3. 数値は Excel 書込前に単位変換される
- 財務数値は決算書単位に合わせて変換
- 株数は 1,000 株単位に丸め

という処理が入ります。

### 10-4. IFRS営業利益は無理に個別値で埋めない
IFRS企業で連結営業利益タグが取れない場合でも、個別 `OperatingIncome` で補完しない運用です。  
空欄のままを正とします。

### 10-5. SellingExpenses は二重加算しない
すでに SellingExpenses が取れている場合、FinancialBusinessCost を足して上書きしません。

### 10-6. 原因調査の順番を固定する
1. `batch_summary.csv`
2. `failed_jobs.csv`
3. `company_jobs.csv`
4. 出力Excelの `raw_edinet`
5. `run.log`
6. それでも分からなければ `DEBUG` で再実行

### 10-7. まず回帰確認する固定会社

#### J-GAAP
- 2206 江崎グリコ株式会社
- 4613 関西ペイント株式会社

#### IFRS
- 7203 トヨタ自動車株式会社
- 6857 株式会社アドバンテスト
- 8001 伊藤忠商事株式会社
- 8473 ＳＢＩホールディングス株式会社
- 9983 株式会社ファーストリテイリング

### 10-8. main では直接作業しない
- 普段の作業継続元ブランチ: `move-to-users-silve`
- 新作業は `move-to-users-silve` から新ブランチを切る
- `main` へ直接コミットしない

## 11. 毎回の基本手順
1. `D:\EDINET_Data\input\zip` に対象 ZIP を入れる
2. `LOG_MODE` が通常は `NORMAL` になっているか確認する
3. `scripts\run_KANPE.bat` で実行する
4. `D:\EDINET_Data\output\実行日時\reports\batch_summary.csv` を開く
5. 必要に応じて `company_jobs.csv` / `failed_jobs.csv` を確認する
6. 数値確認が必要なら出力Excelを開く
7. 数値が怪しい場合は `raw_edinet` を確認する
8. それでも不明なら `DEBUG` に切り替えて再実行する
